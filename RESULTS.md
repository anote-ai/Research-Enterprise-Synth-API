# Results

Consolidated, measured results from Experiments 1–5, the Ablation Study, the 5-seed multi-API
scaling sweep, the private cold-start validation, the 6-API scale-up, and the LLM-as-a-judge
semantic evaluation. Everything here is real — generated by the scripts in `scripts/`, backed by
the raw data in `data/generated/*.json`, and reproducible via the steps in `README.md`. Nothing
below is projected or illustrative. Full narrative, methodology, and caveats are in
`DESIGN_DOC.md` §6–§11; this file is the summary.

**Scale:** 17 APIs touched by the pipeline (3 training: GitHub/Stripe/Slack; 9 real
held-out: Zoom/DigitalOcean/Spotify/Twilio/Notion/OpenAI/Jira/Asana/Trello; 5 private held-out:
hand-authored, never-published synthetic specs), 45–89 examples per experiment, evaluated with a
0.5B fine-tuning target model.

## Experiment 4 — Schema-Based Verification

`scripts/run_experiment4.py` · data: `data/generated/experiment4_verification.json`

45 valid trajectories tested (100% pass), 44 deliberately corrupted variants tested:

| Error type | Detected (final) |
| --- | --- |
| Wrong HTTP method | 12/12 |
| Missing required parameter | 11/11 |
| Invalid endpoint path | 12/12 |
| Parameter type mismatch | 9/9 |
| **Total** | **44/44 (100%)** |

**This took real work to earn.** First run: 57–80% detection (per-API), not 100% — adversarial
testing surfaced four real bugs: (1) verifier accepted any value for declared type `"string"`,
including objects; (2) the corruption harness could drop an optional param instead of a required
one; (3) Stage 1 silently dropped `$ref` parameters (same bug as Experiment 1); (4) Stage 1 never
parsed `requestBody` schema fields into typed parameters at all. All four fixed and
regression-tested.

![Experiment 4 figure](paper/figures/exp4_verification_before_after.png)

**Figure 4.** Invalid-case detection rate by corruption type, first run vs. final. Missing-parameter
and wrong-type detection were the two categories that required real bug fixes to reach 100%.

## Experiment 5 — Downstream LLM Agent Evaluation ⭐

`scripts/run_experiment5.py` · data: `data/generated/experiment5_results.json`

**Hardware-scoped substitute:** the paper's target (Mistral-7B/Llama-3-8B via LoRA) isn't feasible
without a GPU. Substituted Qwen2.5-0.5B-Instruct, actually LoRA fine-tuned on real hardware.

Training set: 45 Stage-6-verified trajectories (GitHub/Stripe/Slack). Held-out eval: 16 intents for
**Zoom** (373 endpoints) — never touched by training or any prior experiment.

| Model | Tool Selection Accuracy | Parameter Validity (of correct) |
| --- | --- | --- |
| Base (zero-shot, untuned) | 12.5% (2/16) | 0.0% |
| + LoRA on Self-Instruct-bootstrapped data | 25.0% (4/16) | 50.0% (2/4) |
| + LoRA on EnterpriseSynth-verified data | **87.5% (14/16)** | **57.1% (8/14)** |

Training loss: 0.708 → 0.403 → 0.247 across 3 epochs (monotonic, real learning).

![Experiment 5 figure](paper/figures/exp5_downstream_performance.png)

**Figure 5.** Left: base vs. LoRA-fine-tuned Qwen2.5-0.5B-Instruct on the held-out Zoom eval set.
Right: training loss across the 3 fine-tuning epochs.

**The parameter-validity gap is itself a finding:** for Zoom's `PUT /users/{userId}/password`, the
real required field is `password`; the fine-tuned model invented `new_password` and
`expiration_time` instead — it generalized *which endpoint to call* but not *the exact field
names* a genuinely unseen schema requires. This is precisely the failure mode Stage 6
verification exists to catch downstream.

**Self-Instruct baseline** (`scripts/run_baseline_selfinstruct.py` +
`run_baseline_selfinstruct_finetune.py`) — the first real evidence for RQ2. Schema-free bootstrap
(4 human-quality seeds, few-shot generation, ROUGE-L dedup), fine-tuned/evaluated with identical
methodology on the identical held-out set. Any fine-tuning helps (12.5%→25%), but
EnterpriseSynth's schema-grounded, verified data helps ~3.5x more (25%→87.5%) — isolating
training-data quality as the source of the gap. Honest surprise: 12/45 (26.7%) of Self-Instruct's
"invented" endpoints turned out real, but **all 12 were GitHub** (branch protection, webhooks) —
the base model's pretraining familiarity with GitHub's public API leaking through, not schema
grounding. Strengthens rather than weakens the cold-start case: truly private internal APIs
wouldn't have this incidental advantage.

**Does 12.5%→87.5% generalize? Scaled to 3 held-out APIs — and the honest answer is no, not
uniformly** (`scripts/scale_experiment5_heldout.py`, data:
`experiment5_multi_api_results.json`). Trained each model once, evaluated against Zoom, plus two
more never-trained-on APIs: DigitalOcean (290 endpoints) and Spotify (88 endpoints).

| Held-out API | Base | Self-Instruct | EnterpriseSynth |
| --- | --- | --- | --- |
| Zoom | 12.5% | 25.0% | 75.0% |
| DigitalOcean | 31.2% | **50.0%** | 43.8% |
| Spotify | 12.5% | 25.0% | 43.8% |

EnterpriseSynth beats Self-Instruct on 2/3 (Zoom, Spotify) but **loses on DigitalOcean** (43.8% vs.
50.0%) *in this single run* — see the 5-seed sweep immediately below for why "single run" matters
here. The retrained Zoom number (75.0%) is also lower than the original single-run 87.5% —
expected variance, since neither weight init nor training-data order is seed-fixed. Parameter
Validity favors EnterpriseSynth more consistently (58.3%/100%/71.4% vs. 50%/100%/25%) — a hint
that verified data may transfer better to argument correctness than tool selection. Unconfirmed
hypothesis for DigitalOcean's reversal: its DevOps-flavored REST conventions may structurally
resemble GitHub's more than Zoom's/Spotify's do, giving Self-Instruct's GitHub-heavy bootstrap an
incidental edge there specifically.

### Is any single seed representative? A 5-seed sweep

`scripts/scale_experiment5_heldout.py` now accepts `--seed N` and reuses the committed held-out
eval sets rather than regenerating them via fresh API calls each run, so a seed sweep varies only
LoRA training randomness, not the eval questions. We reran the full comparison for 5 independent
seeds (42, 123, 777, 2025, 9999) — data: `experiment5_multi_api_results_seed{42,123,777,2025,9999}.json`,
aggregation: `scripts/aggregate_multi_seed_scaling.py` → `experiment5_multi_seed_summary.json`.

| Held-out API | Base | Self-Instruct | EnterpriseSynth |
| --- | --- | --- | --- |
| Zoom | 12.5 ± 0.0% | 23.7 ± 10.3% | **66.2 ± 18.0%** |
| DigitalOcean | 31.2 ± 0.0% | 37.5 ± 21.2% | **53.7 ± 13.7%** |
| Spotify | 12.5 ± 0.0% | 12.5 ± 7.7% | **46.3 ± 7.1%** |

**Averaged over 5 seeds, EnterpriseSynth beats both the base and Self-Instruct on all three APIs,
including DigitalOcean.** The single-run loss reported above was a real draw from a
high-variance distribution (±13.7 for EnterpriseSynth, ±21.2 for Self-Instruct on DigitalOcean,
overlapping ranges), not the central tendency — individual seeds can and do still lose there, so
read this as "wins on average, with real run-to-run variance," not "always wins." Both findings
are reported; neither erases the other.

### Private cold-start validation: never-published enterprise APIs

Every held-out API above is real and well-documented, plausibly present in pretraining data —
leaving the actual cold-start claim untested in its strongest form. We hand-authored 5
never-published, synthetic enterprise specs — CRM, HRIS, Procurement, Ticket Management, Asset
Management (28 endpoints, `data/specs/private/`) — and ran the pipeline against them unmodified.
Scripts: `scripts/generate_private_specs.py`, `build_private_coldstart_eval.py`,
`run_private_coldstart_eval.py`. Data: `private_coldstart_results.json`.

| Eval set | Base (untuned) | EnterpriseSynth-tuned |
| --- | --- | --- |
| Public (Zoom/DigitalOcean/Spotify, n=48) | 18.8% | 39.6% |
| Private (never-published, n=30) | 23.3% | **40.0%** |

**The fine-tuning effect holds on APIs that cannot be in the base model's pretraining data** —
EnterpriseSynth-tuned accuracy on private domains (40.0%) essentially matches public held-out
accuracy (39.6%). The single strongest evidence in this project that the effect is genuine
schema-grounding, not incidental pretraining familiarity. Still one un-seeded run, unlike the
public comparison's 5-seed treatment.

## Semantic Quality Evaluation (LLM-as-a-Judge)

`scripts/run_llm_judge_eval.py`, using `src/enterprisesynth/llm_judge.py` (Claude Sonnet 5 as an
independent judge) · data: `llm_judge_results.json`, `llm_judge_summary.json`

Every result above is Tool Selection Accuracy — binary, endpoint-only. It can't say whether the
call is actually usable. We judged all 48 EnterpriseSynth predictions from one seed-42 run on 4
dimensions (1–5 scale) plus a primary-error classification; 47 of 48 judge responses parsed.

| Dimension (1–5) | Mean |
| --- | --- |
| Intent Match | 2.81 |
| Argument Correctness | 2.19 |
| Missing Parameters | 2.55 |
| Reasoning Quality | 2.28 |

| Error Type | Count | % |
| --- | --- | --- |
| Wrong tool selected | 16 | 34.0% |
| Missing required parameter | 11 | 23.4% |
| Invalid request format | 5 | 10.6% |
| Hallucinated parameter | 3 | 6.4% |
| Incorrect argument value | 2 | 4.3% |

**The headline finding: binary Tool Selection Accuracy overstates practical quality by roughly
2×.** Of the 23 examples marked correct by the binary metric (48.9% of 47), the judge flagged a
real problem in 14 of them (61%) — usually a missing or hallucinated parameter. Only 10/47 (21.3%)
are fully correct by the stricter standard. **Every Tool Selection Accuracy number in this
document should be read as an upper bound on practical correctness, not an estimate of it.**

**A real bug found building this:** the first version used `max_tokens=250`; Claude spent the
whole budget on internal reasoning before writing JSON, silently dropping 17/48 (35%) of judge
calls (`stop_reason: max_tokens`, zero text output). Raising to 1200 cut the failure rate to 1/48
(2%) — reported because it's the same lesson as Experiment 4's bug-discovery story: check an
evaluator's own success rate, don't assume it.

## Ablation Study

`scripts/run_ablation_study.py` (A1/A3/A4) + Experiment 4's data (A2) ·
data: `data/generated/ablation_*.json`

Three ablations proposed in an earlier draft were **dropped and not run**, because the components
don't exist in the codebase: Knowledge Graph (Stage 2 was never built), Planner (combined with
the Trajectory Generator from the start), Response Schema Modeling (only a boolean flag exists).

| Variant | Validity | Diversity | Verification Pass |
| --- | --- | --- | --- |
| Full pipeline (baseline) | 100% | 100% | 100% |
| A1: − Intent Generation | 93.3–100% | 93.3–100% | n/a |
| A2: − Verification | n/a | n/a | **0%** (nothing filtered) |
| A3: + API descriptions | unchanged (ceiling) | unchanged (ceiling); real qualitative diff. | n/a |
| A4: + Full-API context | unchanged (ceiling) | unchanged (ceiling); no valid signal | n/a |

- **A1 (without Intent Generation):** real, if modest, degradation — e.g. Slack's
  `POST /users.profile.set` got an invented, schema-invalid `token` field without intent grounding.
- **A2 (without Verification):** the clearest result in the whole study — 0% vs. 100% of planted
  errors caught.
- **A3 (with descriptions):** quantitative metrics ceiling at 100% both ways; qualitative
  inspection shows a real but modest difference (more explicit semantic grounding, e.g. numeric
  IDs). Reported as inconclusive by the metrics used, not oversold as a positive result.
- **A4 (full-API context):** a "sequencing-language" proxy metric looked promising (6–7
  hits/API) until inspection showed every hit was a false positive. Reported as an invalid metric,
  not kept as a finding.

**A5 — Haiku 4.5 semantic-plausibility check** (`scripts/run_ablation_haiku.py`, data:
`data/generated/ablation_haiku_semantic_check.json`) — this was described in the design doc as an
optional ablation arm but never implemented until an audit
([issue #1](https://github.com/Rashmioffcialpage/enterprisesynth-api/issues/1)) flagged the gap.
Tests whether a cheap LLM catches semantically-wrong-but-structurally-valid values (e.g. a
negated amount, a placeholder string) that the deterministic verifier cannot see by design:

| API | Valid trajectories judged plausible | Semantic corruption still structurally valid | ...caught by Haiku |
| --- | --- | --- | --- |
| GitHub | 10/15 (66.7%) | 15/15 (100%) | 15/15 (100%) |
| Stripe | 15/15 (100%) | 15/15 (100%) | 15/15 (100%) |
| Slack | 13/15 (86.7%) | 15/15 (100%) | 15/15 (100%) |

Confirms the deterministic gate is blind to this error class by construction (100% pass
structurally) and that Haiku catches 100% of them — genuine incremental value. But also a real,
disclosed limitation: GitHub's 33% false-positive rate, traced to Haiku being overly literal (e.g.
flagging correct use of repository IDs as "unverifiable," or expecting a parameter that doesn't
exist in the spec). Adds value for its target error class, but isn't free — would need calibration
before serving as a hard gate rather than an advisory signal.

## Key Findings

- 100% schema-extraction accuracy — trustworthy only because two genuine Stage 1 gaps (`$ref`
  resolution, `requestBody` field parsing) were found and fixed, not assumed.
- Schema-grounded generation produces enterprise-specific, non-generic intents and correctly-scoped
  trajectories on a 45-example pilot.
- Static, non-LLM verification catches 100% of planted errors across four corruption types — but
  only after adversarial testing forced real fixes; the pre-fix rate (57–80%) is reported, not
  hidden.
- EnterpriseSynth-generated SFT data measurably improves tool-selection accuracy on a genuinely
  unseen API (12.5% → 87.5% on Zoom); exact-field-name generalization remains a real, open
  limitation.
- **A real baseline backs this up; averaged over 5 seeds, the effect is uniform after all.**
  Self-Instruct-fine-tuned data reaches 23.7% (Zoom) on average vs. EnterpriseSynth's 66.2% —
  isolating training-data quality as a real factor. A single early run showed EnterpriseSynth
  losing to Self-Instruct on DigitalOcean; the 5-seed sweep shows it winning there on average
  too (53.7% vs. 37.5%), with individual seeds still capable of losing given high variance. Both
  the single-run loss and the 5-seed win are reported — the second contextualizes the first, it
  doesn't erase it.
- **The effect holds on APIs that cannot be in the base model's pretraining data.**
  EnterpriseSynth-tuned accuracy on 5 never-published synthetic specs (40.0%) essentially matches
  public held-out accuracy (39.6%) — the strongest evidence yet that this is genuine
  schema-grounding, not incidental familiarity with well-documented public APIs.
- **Scaling to 6 more real APIs (Twilio, Notion, OpenAI, Jira, Asana, Trello): wins on all 6.**
  17 APIs now touched by the pipeline total, with real training/eval latency data captured as a
  byproduct.
- **Binary Tool Selection Accuracy overstates practical quality by roughly 2×.** An independent
  LLM-as-a-judge evaluation found 61% of "correct" predictions (by the binary metric) still had a
  real defect, usually a missing or hallucinated parameter — only 21.3% are fully correct by the
  stricter standard, vs. 48.9% by binary accuracy alone. Every accuracy number in this document
  should be read as an upper bound on deployment readiness, not an estimate of it.
- Explicit Intent Generation and Schema Verification are both real, load-bearing components (A1,
  A2). A cheap LLM semantic check (A5) adds genuine value for a distinct error class the
  deterministic gate can't see, but has a real, disclosed false-positive rate (33% on GitHub).
  Descriptions and full-API context show no detected effect yet (A3, A4) — an honest null result
  with current metrics, not evidence they don't matter.
